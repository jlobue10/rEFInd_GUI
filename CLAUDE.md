# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Qt (C++17) GUI for customizing and installing the rEFInd bootloader, aimed especially at dual-boot handhelds (ASUS ROG Ally/Ally X, Legion Go, Steam Deck). It builds for **both Linux and Windows** from the same sources. Linux install targets: Fedora/Nobara/Bazzite (RPM) and CachyOS (PKGBUILD); Windows: `windows/install-rEFInd-GUI.ps1` into `%LOCALAPPDATA%\rEFInd_GUI`.

## Build

The only compiled component lives in `GUI/src` (CMake, auto-detects Qt6 then Qt5). Linux:

```
cd GUI/src
mkdir -p build && cd build
cmake ..
make
```

Windows (MSYS2 UCRT64 shell ‚Äî installed at `C:\msys64` on this machine; launch via `MSYSTEM=UCRT64 C:\msys64\usr\bin\bash.exe -lc "..."`):

```
cmake -G Ninja -S GUI/src -B <builddir>
cmake --build <builddir>
windeployqt6 <builddir>/rEFInd_GUI.exe   # plus MinGW/ICU DLLs for standalone use (ldd the exe)
```

**This machine's quirk:** Norton AV silently deletes `CMakeCache.txt` for build dirs inside `Documents` ‚Äî configure into a build dir outside the repo (e.g. the scratchpad). The Windows exe embeds a `requireAdministrator` manifest (`rEFInd_GUI.manifest` via `rEFInd_GUI.rc` for MinGW, link flag for MSVC), so it triggers UAC on launch. Norton also MITMs TLS here; its root CA was exported into MSYS2's trust store (`/etc/pki/ca-trust/source/anchors/norton-root.pem`) ‚Äî if pacman ever fails with certificate errors again, that's why, and `curl --ssl-no-revoke` is how to confirm a URL works past Norton's revocation break.

### Windows packaging

`windows/assemble-deploy.sh <exe> deploy` gathers the exe + Qt plugins (windeployqt) + the MinGW/ICU DLL closure (`windows/copydeps.sh`, an `ldd`-based best-effort copy) + `windows/*.ps1` + `icons/` + `backgrounds/` + a seed `GUI/refind.conf` into `deploy/` ‚Äî the exact final runtime layout. `windows/rEFInd_GUI.iss` (Inno Setup) packages `deploy/` into a **per-user** installer targeting `%LOCALAPPDATA%\rEFInd_GUI`, which equals `Platform::dataDir()`, so install dir and runtime data dir are the same directory (no Program Files, no admin needed to install; the app elevates itself). `.github/workflows/windows-release.yml` runs the whole chain on a `v*` tag, and Authenticode-signs the exe, the `.ps1` scripts, and the installer via **SignPath Foundation** (free OSS signing) ‚Äî two-stage: sign `deploy/` contents, then build and sign the installer. Signing is gated on the `SIGNPATH_ORGANIZATION_ID` repo variable, so the workflow still produces unsigned builds until SignPath is configured (`windows/SIGNING.md`). Build artifacts (`deploy/`, `build-*/`, `windows/Output/`) are gitignored.

There are no tests and no linter.

**Important:** both `rEFInd_GUI.spec` (`%prep`) and `PKGBUILD` (`build()`) do a fresh `git clone` of this repo from GitHub rather than using the local checkout. Package builds therefore only pick up changes after they are pushed to `main`.

## Releasing / version bumps

The version is duplicated in several places that must be kept in sync:

- `VERSION` (plain text, e.g. `2.0.0`) ‚Äî fetched at runtime by the "Check For Update" button and compared component-wise (`QVersionNumber`) against `APP_VERSION`
- `GUI/src/mainwindow.cpp`: `static const char APP_VERSION[]` (also feeds the About box text)
- `GUI/src/CMakeLists.txt`: `project(rEFInd_GUI VERSION ...)`
- `GUI/src/rEFInd_GUI.manifest`: `assemblyIdentity version="X.Y.Z.0"` (four-part)
- `windows/rEFInd_GUI.iss`: `#define AppVersion`
- `rEFInd_GUI.spec`: `Version:` (plus a `%changelog` entry)
- `PKGBUILD`: `pkgver`
- `debian/changelog`: new top entry (`refind-gui (X.Y.Z) unstable; ...`)

## Architecture

Platform-specific code is confined to three files; everything else is shared. Never build shell command strings from user input ‚Äî external commands go through `QProcess` with argument lists, file copies through `QFile::copy`.

- **`OSDetector`** (interface in `osdetect.h`): `osdetect_common.cpp` holds the platform-neutral logic. `detect()` iterates **every** ESP in the partition list and scans each reachable one (`espScanRoot(part)` ‚Äî platform-specific), so multi-ESP machines (Windows ESP on one disk, Linux ESP on another) are fully covered rather than just one "system" ESP. Every discovered entry is tagged with its ESP's **partition GUID** as `volume` (`espVolumeId`), so the generated `menuentry` boots regardless of which ESP rEFInd itself launched from. The per-ESP `EFI/` vendor-dir scan (loader preference shim ‚Üí GRUB ‚Üí systemd-boot; `bootmgfw.efi` ‚Üí Windows, `steamcl.efi` ‚Üí SteamOS) also renames a bare systemd-boot loader (`/EFI/systemd`) to the running distro when it's on that distro's own ESP (so CachyOS shows "CachyOS", not "systemd-boot"); on any other ESP (e.g. a Linux ESP scanned from Windows, where os-release doesn't exist) it falls back to the `title` lines in that ESP's `loader/entries/*.conf`, mapping any title word onto the known-distro-name table ("Linux Cachyos Deckify" ‚Üí "CachyOS"). Cross-volume rules in `assembleEntries` cover ESPs that couldn't be scanned (unmounted `SYSTEM`/`SYSTEM_DRV`-labelled Windows ESP ‚Üí Windows by partition GUID; `BATOCERA`/`VTOYEFI`; removable ESPs by partition GUID). `osdetect_linux.cpp` enumerates via `lsblk -J` ‚Äî **`NAME` must be in the `-o` column list** or lsblk (‚â• 2.42) drops the disk‚Üíchildren tree and emits partitions flat; the parser walks recursively so either shape works. `MOUNTPOINT` is the pre-2.37 util-linux fallback for `MOUNTPOINTS`. It names the running distro from `/etc/os-release` `ID`/`ID_LIKE` (so `/EFI/fedora` shows "Nobara" on Nobara); an ESP mounted at `/boot`, `/boot/efi`, or `/efi` is the running system's. On Linux an ESP is scannable only where already mounted (mounting needs root, which detection avoids). `osdetect_win.cpp` enumerates via one PowerShell `Get-Partition`/`Get-Disk`/`Get-Volume` ‚Üí JSON call (including each partition's `\\?\Volume{guid}\` access path) and mounts the letterless system ESP with `mountvol <X>: /S`; a letterless **non-system** ESP (e.g. a Linux distro's ESP ‚Äî the ROG Xbox Ally X dual-boot case) is mounted on a temp *directory* via `mountvol <dir> \\?\Volume{guid}\`, which doesn't consume a drive letter (both unmounted after; `releaseEspRoot` treats any root longer than `"Z:"` as a directory mount point to remove). Without that mount the Windows GUI never saw CachyOS's systemd-boot at all. Windows gotcha: letterless partitions serialize `DriveLetter` as a NUL char, not empty ‚Äî hence the "is it actually a letter" check.
- **`Platform` namespace** (`platform.h/.cpp`, the only `#ifdef Q_OS_WIN` site): data dir (`~/.local/rEFInd_GUI` vs `%LOCALAPPDATA%\rEFInd_GUI` ‚Äî same layout below it), installer/config-install/randomizer launches (xterm + .sh + systemd vs PowerShell + .ps1 + Scheduled Task), firmware_bootnum availability (Linux only; the checkbox is disabled on Windows), Install Source options (Windows has only "Sourceforge"). `installConfig()` runs the install script synchronously on both platforms (Windows with `CREATE_NO_WINDOW`), captures its combined output, and `MainWindow` shows the success/failure dialog from that output ‚Äî the scripts print short human-readable messages and distinct exit codes for exactly this reason.
- **Unreadable-ESP detection recovery + Deep Scan** (since 2.4.0, ported from the sibling SteamDeck_rEFInd repo ‚Äî the four `osdetect_*` files are kept **byte-identical** to that repo's except its extra `/esp` mountpoint in `isRunningSystemEsp()`; port detection fixes by copying the files and re-applying that one hunk there, not here). Detection runs unprivileged, and Fedora-family installs mount `/boot/efi` `umask=0077` root:root, so `scanEspRoot()` used to silently see nothing. `detect()` now distinguishes "couldn't look" (`espRootUnreadable()`) and recovers entries from the cache written by `scan_esp.sh` (an opt-in elevated scan behind the Deep Scan button, staged to `~/.local/rEFInd_GUI/scan_esp.sh`, cache at `<dataDir>/GUI/esp_scan.conf` keyed by partition GUID) or, without a cache, from `efibootmgr -v` (world-readable, no password). `classifyLoaderPath()` names recovered entries exactly as a filesystem scan would and mirrors `scanEspRoot()`'s skip list (rEFInd, `/EFI/BOOT`, drivers/tools/memtest, non-`bootmgfw` Microsoft plumbing). The Deep Scan button is disabled via `espDeepScanUseful()` when every conventional ESP mountpoint is readable. **systemd automounts** (since 2.6.1): SteamOS 3.9 mounts `/esp` and `/efi` through `esp.automount`/`efi.automount` ‚Äî until a path resolves *through* the mount point, the vfat mount doesn't exist, lsblk shows the partition unmounted, and detection skipped the ESP entirely (only label-matched media like Ventoy survived; Deep Scan reported "No ESP found"). Three layers fix this: `listPartitions()` and `espDeepScanUseful()` stat `<point>/.` for `/esp`, `/efi`, `/boot/efi`, `/boot` to trigger the automounts (a plain `stat()` of the mount point does NOT trigger ‚Äî `AT_NO_AUTOMOUNT` since Linux 4.14; the trailing `/.` forces traversal, which triggers regardless of the root-only permission on the mounted fs); `detect()` recovers entries for a *completely unmounted* ESP from the deep-scan cache / `efibootmgr` by partition GUID (previously only mounted-but-unreadable ESPs got recovery); and `scan_esp.sh` passes unmounted ESPs to its root payload as device paths, which temp-mounts them read-only for the scan. Also beware findmnt on automounted paths: it lists the `autofs` row (SOURCE `systemd-1`) **before** the real vfat row, so `findmnt -no SOURCE <mp> | head -1` returns `systemd-1` ‚Äî the scripts use `grep -m1 '^/dev/'` for SOURCE and `tail -1` for FSTYPE instead; `findmnt -S <dev>` is immune (autofs rows have a different source device).
- **`MainWindow`** (`mainwindow.cpp`, platform-free): detection results feed the four Boot Option combos (`BootEntry` payloads via `QVariant`), refreshed by the Rescan button. `on_Create_Config_clicked` ‚Üí `createBootStanza` writes `<dataDir>/GUI/refind.conf`, one data-driven stanza per non-None slot; `default_selection` is the chosen entry's position among generated stanzas. Legion Go (DMI/CIM board `LNVNB161216`) forces `resolution 2560 1600`; Legion Go 2 (DMI product name `83N0`/`83N1` ‚Äî same board name as the Go 1, so it's checked first) forces `resolution 1920 1200`; ROG Xbox Ally / Ally X (DMI/CIM board prefix `RC73YA`/`RC73XA`) forces `resolution 1920 1080` (the numbered `resolution 3` picks the wrong mode there). Devices with no quirk get `resolution <W> <H>` from `OSDetector::nativePanelResolution()` ‚Äî the built-in panel's EDID/DRM preferred mode, internal display (eDP/LVDS/DSI, or WMI `VideoOutputTechnology` INTERNAL/eDP/LVDS on Windows) preferred over external outputs so a docked handheld reports its own screen; Linux reads the first line of `/sys/class/drm/card*-*/modes`, Windows parses the EDID preferred-timing DTD via `WmiGetMonitorRawEEdidV1Block`. Note it returns the mode **as the panel reports it** (portrait on portrait-native panels ‚Äî that's what the Legion Go quirks exist to override). Only when nothing is detected does the config fall back to `resolution 3`. Selected PNGs are copied to canonical names (`background.png`, `os_icon1..4.png`).
- **rEFInd install scripts, efibootmgr phase** (`refind_install_Sourceforge.sh`, `refind_install_package_mgr.sh`): the installer-created "rEFInd Boot Manager" entry (exact label) is deleted first, then the new `rEFInd` NVRAM entry is created **before** any plain old rEFInd entries are deleted ‚Äî the old delete-then-create order left machines with *no* rEFInd entry whenever the create failed (observed live: `lsblk -no PKNAME` returned empty under util-linux 2.42, producing `efibootmgr -c -d /dev/ ...`). Disk resolution falls back to sysfs (`/sys/class/block/<part>` parent dir) when PKNAME is empty and validates `-b "$ESP_DISK"` before touching NVRAM. The new entry is identified as the head of BootOrder (which `efibootmgr -c` sets) so cleanup can't delete it. efibootmgr ‚â• 18 appends a tab + device path after the label even without `-v`, so every label regex must end `(\t.*)?$` rather than anchoring at the label ‚Äî `$`-anchored label matches silently never match (observed live: neither the "rEFInd Boot Manager" pre-delete nor the new-entry check matched, leaving two rEFInd entries). Both scripts end with a verification summary read back from live NVRAM (entry exists + first in BootOrder) and keep their xterm open (`read` prompt, tty-guarded) so status is visible; the package-mgr script must keep zenity's stdout protocol clean ‚Äî diagnostics go to stderr. After the efibootmgr phase, both scripts run `sudo sbctl-batch-sign` when the OS is CachyOS (`/etc/os-release` `ID=cachyos`) **and** Secure Boot is enabled (SecureBoot efivar's fifth byte = 1, dependency-free `od` read) ‚Äî CachyOS enforces sbctl-signed binaries under Secure Boot, so unsigned rEFInd files would be refused by the firmware; missing tool or a failed sign warns and continues. `windows/install_rEFInd.ps1` mirrors the applicable parts: it writes a dedicated `rEFInd` **Boot#### EFI variable directly** (`SetFirmwareEnvironmentVariableW` + `SeSystemEnvironmentPrivilege`, byte-identical to what `efibootmgr -c` produces ‚Äî the EFI_LOAD_OPTION builder was validated byte-for-byte against a live efibootmgr-created entry) and prepends it to `BootOrder`. It must NOT repoint `{bootmgr}` as versions ‚â§2.2.0 did: the Windows Boot Manager entry carries a `WINDOWS...BCDOBJECT={...}` optional-data blob that trails the loader path in `efibootmgr -v` ("malformed entry", issue #23) and reaches rEFInd as junk load options ‚Äî the installer now restores a repointed `{bootmgr}` when it finds one (migration). bcdedit **cannot** create firmware entries on current Windows 11 (`/application firmware` was removed; adding a non-firmware object to `{fwbootmgr} displayorder` reports success but is silently ignored), hence the direct NVRAM write. Entries carrying the `"WINDOWS"` blob (hex `57494e444f5753`) are never overwritten/deleted ‚Äî that's `{bootmgr}`'s own variable. `bcdedit` (via the exit-code-checked `Invoke-Bcdedit` wrapper ‚Äî a native exe whose failures don't trip `$ErrorActionPreference='Stop'`, and stderr under `2>&1` becomes a terminating NativeCommandError) is still used for `{bootmgr}` migration/snapshot; state is backed up before modification and the script ends with a read-back SUCCESS/WARNING/FAILED summary from live NVRAM. Progress is shown as numbered colored step banners (`Write-Step`, 6 steps) plus a `Write-Progress` bar, with the summary verdict colorized. If Secure Boot is enabled (`Confirm-SecureBootUEFI`, try/catch = "not enabled"), a yellow note explains that sbctl-managed distros (CachyOS) must sign the new binaries from Linux ‚Äî **rEFInd_GUI only; the SteamDeck_rEFInd copy of this script deliberately omits the Secure Boot block**, and is otherwise identical modulo the `rEFInd_GUI`‚Üí`SteamDeck_rEFInd` data-dir substitution. The script ends with a tty-style `Read-Host` pause then `[Environment]::Exit`, which closes the window even under the launcher's `-NoExit` (no stray prompt). `windows/uninstall_rEFInd.ps1` reverses everything (restore `{bootmgr}` if repointed, delete rEFInd Boot#### entries **whose HD node matches the system ESP GUID** ‚Äî rEFInd entries on other ESPs, i.e. Linux-side installs, are reported and left alone ‚Äî remove `EFI\refind`/`EFI\Xbox360`, unregister the randomizer task; `-KeepEspFiles` keeps files); the Inno uninstaller runs it elevated via `[UninstallRun]`+`Verb: runas` after a yes/no prompt (`InitializeUninstall`), and `[UninstallDelete]` scrubs the whole per-user app dir. The UEFI NVRAM helper block is duplicated between install and uninstall scripts ‚Äî keep in sync.
- **Privileged actions, Linux**: rEFInd install scripts run in `xterm`; "Install Config" runs `sudo -n /etc/rEFInd/install_config_from_GUI.sh`, passwordless via the sudoers entry (`zz_install_config_from_GUI` ‚Üí `/etc/sudoers.d/zz_install_config_from_GUI`, root-owned 0440, visudo-gated; the `zz_` prefix is load-bearing ‚Äî sudo applies the last matching entry in lexical sudoers.d order, so a passworded catch-all drop-in like SteamOS's `wheel` would override a file sorting before it; the installer also removes the old-name rule from previous versions) that `install-rEFInd-GUI.sh` installs. Since 2.6.2 the GUI refuses to run that script unless it SHA-256-matches the copy embedded in the binary at build time (`Platform::installConfigScriptTrusted`; the repo script is a Qt resource in `resources.qrc`, with the installer's `HOME` sed replicated before hashing) ‚Äî a mismatch pops a tamper warning suggesting reinstall instead of running it. Consequence: any edit to `install_config_from_GUI.sh` requires a rebuilt/re-released GUI binary, or every user's Install Config is blocked as "modified". Windows deliberately skips the check (Authenticode signing rewrites the `.ps1` files, so a build-time hash can't match). **Windows**: the exe itself runs elevated (manifest), and actions run the `windows/*.ps1` scripts from the data dir; `install_rEFInd.ps1` repoints the `{bootmgr}` firmware entry via `bcdedit` (backup + revert command printed) ‚Äî never run it casually on a dev machine.
- **Config install targets rEFInd's real ESP**, not just the conventionally-mounted one ‚Äî critical on multi-ESP machines where rEFInd lives on the Windows ESP while the running distro mounts its own ESP at `/boot`. `install_config_from_GUI.sh` reads the rEFInd entry's partition GUID from `efibootmgr -v` (`HD(...,GPT,<guid>,...)`), resolves it via `blkid`/`lsblk`, and mounts it on a temp dir if it isn't already mounted (EXIT-trap unmount); it falls back to the old `/boot/efi ‚Üí /efi ‚Üí /boot` findmnt heuristic when there's no efibootmgr/NVRAM entry. `windows/install_config_from_GUI.ps1` mirrors that NVRAM-first logic: it parses `bcdedit /enum firmware` for entries (including a repointed `{bootmgr}`) whose `path` is `\EFI\refind\refind*.efi` (or, mirroring the Linux script's second-tier match, an entry `bcdedit` shows with `description rEFInd` ‚Äî the label a Linux `efibootmgr -c -L rEFInd` install sets ‚Äî for when bcdedit renders that entry's path in a form the strict regex misses), takes the one earliest in `{fwbootmgr}` displayorder, and maps its `\Device\HarddiskVolumeN` device to a partition via `QueryDosDevice` over each ESP's `\\?\Volume{guid}` name ‚Äî a "contains rEFInd" filesystem scan (system ESP first) is only the fallback. This matters on multi-ESP machines where a **stale** `EFI\refind` on the Windows ESP would otherwise shadow the live one on the Linux ESP that firmware really boots (observed on the ROG Xbox Ally X: config went to the Windows ESP while the NVRAM rEFInd entry booted the CachyOS ESP). `windows/rEFInd_bg_randomizer.ps1` duplicates the same NVRAM-first selection (these scripts are deliberately standalone); keep them in sync.
- **Settings persistence**: `QSettings` INI at `<dataDir>/GUI/rEFInd_GUI.ini`, read on construct / written on destruct. Boot-option selections are stored by *text* (`BootOption0XText`), not index ‚Äî combo contents are dynamic.

## Internationalization (GUI)

The GUI is translated via Qt Linguist files: `GUI/src/rEFInd_GUI_{en_US,ar,bn,de,es,fa,fr,hi,id,it,ja,ko,pt,ru,tr,uk,ur,vi,zh_CN}.ts`, listed in `TS_FILES` in `CMakeLists.txt`. On Qt 6, `qt_add_translations()` compiles them at build time and embeds the `.qm` under `:/i18n`, where `main.cpp` loads them with the locale-fallback `QTranslator::load(QLocale(), ...)` overload (plus a `qtbase` translator for Qt's own dialog strings); on Qt 5 the `.qm` are only generated into `translations/` next to the binary. Rules: wrap every new user-facing GUI string in `tr()` (in `Platform`, use `QCoreApplication::translate("Platform", ...)`); after changing strings, refresh **all** `.ts` files with `lupdate` (or the `update_translations` target) so translators see the new entries ‚Äî unfinished entries just fall back to English. Do NOT translate `BootEntry` display/menu names ("Windows", "SteamOS", "Ventoy", ...): they are matching identifiers persisted by text and produced by the parity-locked `osdetect_*` files. `.ui` strings that are overwritten at runtime are marked `notr="true"` ‚Äî keep doing that for new ones. The en_US file is the source-language reference and stays untranslated. The `QT_LAYOUT_DIRECTION` message (anchored in `main.cpp`) must stay `RTL` in the ar/fa/ur files ‚Äî it is what mirrors the layout for right-to-left languages. The bash/PowerShell scripts are deliberately English-only for now (tamper-hash + cross-repo parity constraints) ‚Äî see `I18N_AUDIT.md` before localizing them.

## Install-time file layout (context for editing the shell scripts)

`install-rEFInd-GUI.sh` is the user-facing entry point (curl | sh). It copies `GUI/`, `icons/`, `backgrounds/`, and the install/uninstall scripts to `~/.local/rEFInd_GUI/`, builds/installs the RPM (or rpm-ostree layers a release RPM on Bazzite, or `makepkg -si` on CachyOS), and installs root-owned pieces to `/etc/rEFInd/` (the built binary, `install_config_from_GUI.sh`, `rEFInd_bg_randomizer.sh`).

`uninstall_rEFInd.sh` (repo root, run via sudo) is the Linux counterpart of `windows/uninstall_rEFInd.ps1` ‚Äî keep their behavior in parity: delete the rEFInd boot entries whose partition GUID matches this distro's ESP (loader-path match covers both the `rEFInd` and refind-install's `rEFInd Boot Manager` labels; entries on other ESPs ‚Äî a Windows-side install ‚Äî are reported and left alone), re-activate the Windows boot entry that `refind_install_*.sh` deactivated with `efibootmgr -A`, remove `EFI/refind`, `EFI/Xbox360`, and `/boot/refind_linux.conf` (`--keep-esp-files` skips), disable the randomizer service, and with `--remove-app` also remove the package (dnf/pacman/apt/rpm-ostree), `~/.local/rEFInd_GUI`, `/etc/rEFInd`, the sudoers rule, and desktop entries. It resolves the invoking user's home via `SUDO_USER` (under sudo, `$HOME` is root's).

The literal tokens `USER` and `HOME` inside `zz_install_config_from_GUI`, `install_config_from_GUI.sh`, `rEFInd_GUI.desktop`, and `rEFInd_bg_randomizer.sh` are placeholders replaced by `sed` during installation ‚Äî don't "fix" them to `$USER`/`$HOME`.

**Xbox 360 controller driver**: all three rEFInd install scripts (`refind_install_Sourceforge.sh`, `refind_install_package_mgr.sh`, `windows/install_rEFInd.ps1`) download `UsbXbox360Dxe.efi` into the ESP's `EFI/refind/drivers_x64/` so gamepads work in the boot menu. The URL currently points at the **jlobue10 fork** of the SkorionOS driver (adds Legion Go 2 PIDs + Ally poll-timeout fix + v1.7.3's right-stick-in-Mouse-mode fix, confirmed on a Legion Go 2 in issue #23); switch it back to SkorionOS in all three scripts once upstream PR #7 ‚Äî which supersedes the old PIDs-only PR #6 and carries the full fork state ‚Äî is merged and released upstream. The scripts fetch the standard release `UsbXbox360Dxe.efi` ‚Äî that's the correct default; `UsbXbox360Dxe-debug.efi` (ESP file logging) is for troubleshooting only. The URL is version-agnostic (`.../releases/latest/download/UsbXbox360Dxe.efi`) so no version needs syncing; only the `.efi` is copied (the driver self-creates `\EFI\Xbox360\config.ini` on first boot). Download failure is non-fatal ‚Äî it warns and continues. The package-manager script is the one that must `mkdir -p .../drivers_x64` first, since `refind-install` doesn't populate it. Keep this step in sync across all three scripts when editing.

**Touchscreen driver** (since 2.3.6, generalized from AllyTouchI2cDxe): the same three scripts also download [`TouchI2cDxe.efi`](https://github.com/jlobue10/TouchI2cDxe) into `drivers_x64/`, but **only on devices with a supported HID-over-I2C touch panel** ‚Äî ROG Xbox Ally / Ally X (DMI board_name prefix `RC73XA`/`RC73YA` on Linux, `Win32_BaseBoard.Product` on Windows ‚Äî same board convention the GUI's resolution quirk uses) and Steam Deck OLED/LCD (DMI product_name `Galileo`/`Jupiter` on Linux, `Win32_ComputerSystemProduct.Name` on Windows; LCD supported since driver v1.2.0). These touchscreens are HID-over-I2C (Novatek on the Ally, FocalTech FTS3528 on both Decks ‚Äî I2C-identical on the two models, differing only in the panel reset GPIO, which the driver resolves itself via SMBIOS), invisible to the USB driver; TouchI2cDxe produces `EFI_ABSOLUTE_POINTER_PROTOCOL`, which rEFInd consumes natively (confirmed working on an Xbox Ally X as AllyTouchI2cDxe v1.0.0; the rename to TouchI2cDxe added the Deck profiles). A successful download also deletes a stale `AllyTouchI2cDxe.efi` from `drivers_x64/` ‚Äî the old driver would otherwise load as a second AbsolutePointer producer for the same panel. Same rules as the controller driver: version-agnostic `releases/latest` URL, download failure warns and continues (and keeps any old AllyTouchI2cDxe.efi), no config file needed, removed with `EFI/refind` on uninstall, auto-signed by the CachyOS `sbctl-batch-sign` step (`sbctl verify` scans the whole ESP). In `install_rEFInd.ps1` the extra download is a numbered `Write-Step`, so `$TotalSteps` is 7 on a touch device and 6 elsewhere ‚Äî keep that arithmetic in mind when adding steps. Keep all three scripts in sync.

`rEFInd_bg_randomizer.service` + `rEFInd_bg_randomizer.sh` implement the optional random-background-on-boot feature, toggled from the GUI via `systemctl enable/disable`.
